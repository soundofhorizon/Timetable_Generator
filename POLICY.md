

# POLICY.md

---

## Goal / Non-goals

### Goal
AI（Codex / ChatGPT）に実装を委任しても、

- 設計把握が崩れない
- 差分が読める
- 変更が変に散らない
- コメント（理由・落とし穴）が失われない
- 同期や責務の事故が起きない

状態で、安全に開発が継続できること。

**守る理由:**  
暴走や設計崩壊を防ぎ、「止まらない開発」を実現するため。

---

### Non-goals

- 完璧な設計の追求
- 一発で最適解を出すこと
- 汎用性の最大化
- 抽象化の美しさ

**守る理由:**  
最適化よりも「壊れずに前進すること」を優先する。

---

## Default Principles（デフォルト憲法）

毎回議論しない固定方針。

1. **最小変更優先（Minimal Diff First）**
2. **既存構造を尊重する**
3. **同期を増やさない設計を優先**
4. **基盤層（init / registry / common）にロジックを入れない**
5. **client / server を絶対に混ぜない**
6. **既存型（Optional等）を優先**
7. **例外は握り潰さない**
8. **コメントは資産。削らない。書き足す。**

**守る理由:**  
意思決定回数を減らし、暴走の可能性を固定的に下げるため。

---

## Change Scope Rules（変更範囲・ファイル散り防止）

### 原則

- 1機能 = 3〜7ファイル以内
- 無関係なファイルは触らない
- 横断的修正は禁止（リファクタ専用PRに分離）
- 新規ヘルパー作成は最後の手段

### NG（S1〜S4）

- S1: 関係薄いファイルを大量に触る
- S2: 初期化 / Registry / 共通基盤にロジック侵入
- S3: client/server責務混在
- S4: 類似ヘルパー増殖

### AIルール

AIは実装前に：

- 変更予定ファイル一覧を提示
- 各ファイルの変更理由を1行で説明

許可なしに範囲拡張禁止。

**守る理由:**  
「変な散り方」による設計崩壊を防ぐ。

---

## Comment & Knowledge Preservation

### 絶対原則

- 既存コメントは削除禁止（明確な誤り以外）
- 理由・制約・落とし穴はコードの近くに残す
- 「なぜ」を必ず書く

### 書くべき内容

- なぜこの設計か
- 他案を採用しなかった理由
- 将来壊れやすい点
- 同期や責務の制約

### 禁止

- コメントの言い換え削除
- TODOの削除
- 無言リファクタ

**守る理由:**  
知識が失われると設計把握が崩れるため。

---

## Client/Server Boundary Rules（Fabric境界）

### 原則

- ロジックは基本server側
- clientは描画・入力のみ
- World参照は原則server

### 禁止

- clientでゲームロジック決定
- server専用クラスをclient参照
- 共通コードにclient依存を書く

### AI手順

実装前に：

- 「これはserver責務 / client責務」の宣言を行う

**守る理由:**  
責務混在は修復コストが極めて高い。

---

## State & Sync Rules

### 第一原則

**同期しない設計を最優先。**

優先順位：

1. 計算で再現できるなら再計算
2. serverのみで完結させる
3. クライアントは表示のみ
4. どうしても必要なら同期

### 同期が必要な場合

- 何を同期するか
- いつ同期するか
- 一方向か双方向か
- 破損時の挙動

を事前明示。

### 禁止

- 毎tick同期
- 暗黙同期
- なんとなくPacket追加

**守る理由:**  
同期は最も壊れやすく、影響範囲が読みにくい。

---

## Error & Exception Handling

### 絶対禁止

- catchして握り潰す
- ログなし無視

### 分類軸（recoverable）

| 種類 | 方針 |
|------|------|
| Recoverable | ログ出力 + 安全復帰 |
| 不整合 | 明示ログ + フェイルセーフ |
| 設計違反 | 例外投げる |

### ログ方針

- 「何が」「なぜ」「どの状態で」失敗したか
- 1行ログは禁止（情報不足）

**守る理由:**  
隠れた不具合が最も設計を壊す。

---

## Types & API Design

### 原則

- 既存型優先（Optional / Result的パターン）
- null回避
- 新規型は最小限

### 新規型を作る条件

- コンパイル時に誤用を防げる
- 状態制約を明確化できる
- 同期事故を減らせる

それ以外は禁止。

**守る理由:**  
型増殖は理解コストを増やす。

---

## Naming Rules

### 原則

- 責務が名前に出る
- Manager濫用禁止
- Util増殖禁止

### 仮名許容（C4）

- 迷ったら仮名で進める
- 後で専用PRで整理

**守る理由:**  
命名で詰まって止まるのを防ぐ。

---

## Decision Load Management

### AIは必ず二択〜三択で提示

例：

- A: 既存拡張（変更少）
- B: 新クラス追加（安全性高）
- C: 暫定実装（動作優先）

### 採点形式

- 変更範囲
- 将来負債
- 同期リスク
- 実装コスト

### 妥協ルール

- C1: スコープ削減
- C2: とりあえず動く優先
- C4: 命名・整理は後回し

**守る理由:**  
F1〜F4の詰まりを回避する。

---

## Execution Protocol（AI作業手順）

### 1. 提案

- 目的整理（1段落）
- 選択肢（2〜3）
- 推奨案
- 変更ファイル一覧

### 2. 許可

- ユーザーが案を選択

### 3. 実装

- 宣言したファイルのみ変更
- コメント保持
- 理由をコード近くに記述

### 4. 検証

- 影響範囲説明
- 同期確認
- client/server確認

### 5. 要約

- 何を変えたか
- なぜその設計か
- 将来の注意点

**守る理由:**  
無断実装と範囲拡大を防ぐ。

---

## Quality Gates

### 差分

- 変更ファイル数 ≤ 7
- 無関係ファイル変更なし
- コメント削除なし

### 構造

- client/server混在なし
- 基盤層への侵入なし
- 新規ヘルパー増殖なし

### 同期

- 新規Packet追加時は理由必須
- tick同期禁止

### 例外

- 握り潰しゼロ
- ログ明示

**守る理由:**  
自動的に暴走を検知するため。

# AIに与える指示テンプレ

---

## ① 提案のみ（編集禁止）

```
あなたはPOLICY.mdに従って提案のみ行ってください。
コード編集は禁止です。

目的:
[ここに目的]

制約:
- 変更範囲を最小化
- 同期は可能な限り避ける
- client/server責務を明示する

出力形式:
1. 状況整理（簡潔）
2. 選択肢（2〜3）
3. 推奨案
4. 変更予定ファイル一覧（理由付き）
```

---

## ② 実装（ホワイトリスト付き）

```
あなたはPOLICY.mdに従って実装してください。

目的:
[目的]

変更可能ファイル（この中のみ）:
- A.java
- B.java

禁止:
- 上記以外のファイル変更
- コメント削除
- 同期追加（明示許可なし）

手順:
- まず変更内容の要約を書く
- その後コードを提示
- 最後に影響範囲とリスクを書く
```

---

## ③ バグ修正（ログ入力前提）

```
あなたはPOLICY.mdに従ってバグ修正を行います。

前提:
- ログを読み、原因を仮説化する
- 握り潰しは禁止

入力:
- エラーログ
- 関連クラス

出力:
1. 原因仮説（2〜3）
2. 最有力案
3. 修正方針
4. 変更ファイル一覧
5. 修正コード
6. 再発防止コメント案
```

## Repo Planning Files（GOALS.md / TASKS.md）

### GOALS.md（Goal Backlog）
- 形式はカンバン: Inbox / Next / Doing / Waiting / Done
- 制限: Next <= 3, Doing <= 1
- 各Goalは次を含む:
    - Why
    - Done（観測可能な達成条件）
    - Scope（大雑把でOK）
    - Notes (why/pitfalls)
    - Tasks（紐づくTxxx）
    - Commit/Refs（任意）

### TASKS.md（Task Board）
- 形式はカンバン: Inbox / Next / Doing / Waiting / Done
- 制限: Next <= 3, Doing <= 1
- 各Taskは次を含む:
    - Done（観測可能な完了条件）
    - Entry（入口ファイル/パス）
    - Scope（触ってよいファイル/ディレクトリ）
    - Notes (why/pitfalls)（理由・落とし穴。既存は削除禁止）
    - Commit（完了コミットhash）

**守る理由:**
- 選択肢（意思決定）を減らす
- 変更の散り方を抑える
- 理由/落とし穴を失わない
